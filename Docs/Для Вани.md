# AI / RAG / Impact Engine / LLM Gateway

## Ультра-краткое описание проекта

RegRadar — AI-система, которая объясняет регуляторные изменения, оценивает их влияние и показывает, каких клиентов банка они затрагивают.

## Краткое summary для исполнителя

Твоя зона — весь интеллектуальный контур: LLM gateway через PolzaAI, structured extraction, RAG с источниками, impact score, matching документов с профилями клиентов и генерация уведомлений. Главный принцип: AI не должен галлюцинировать, все выводы должны опираться на документы и сохраняться в структурированном виде.

## Важная сноска для программиста

Работать в **spec-ops подходе для вайб-кодинга**: сначала описать AI-контракты, JSON-схемы, промпт-версии, тест-кейсы и acceptance-критерии, потом генерировать код. Обязательно подключить **MiMo V2.5 / MiMo V2.5-Pro** или совместимую MiMo V2.5-модель как coding-agent для анализа, генерации и ревью. Для продуктовых AI-вызовов использовать gateway-архитектуру: PolzaAI обязателен как основной провайдер-шлюз, но модель не хардкодить.

## Главная цель

Сделать AI-контур, который позволяет юристу банка:

1. открыть карточку изменения;
    
2. увидеть краткое summary;
    
3. увидеть impact score;
    
4. понять, почему score высокий;
    
5. увидеть релевантных клиентов;
    
6. задать вопрос в RAG-чате;
    
7. получить ответ только по базе документов;
    
8. сформировать клиентское уведомление без юридической консультации.

## LLM Gateway

Сделать единый интерфейс:

```md
LLMProvider
→ PolzaAIProvider
→ MockProvider
→ OptionalProvider
```

Требования:

- model выбирается через config/env;
    
- provider не размазан по коду;
    
- все вызовы логируются;
    
- есть retry;
    
- есть timeout;
    
- есть fallback/mock;
    
- промпты версионируются;
    
- результат валидируется по схеме.

## Impact Engine

Сделать гибридный scoring:

```md
impact_score =
regulator_weight
+ topic_weight
+ penalty_weight
+ affected_client_weight
+ process_change_weight
+ urgency_weight
+ confidence_weight
```

Шкала:

- 0–30: low
    
- 31–60: medium
    
- 61–80: high
    
- 81–100: critical
    

Важно:

- score должен быть объяснимым;
    
- рядом показывать причины;
    
- не обещать точную юридическую оценку;
    
- последствия формулировать осторожно;
    
- если штрафы/санкции не найдены в источнике, не выдумывать.
    

## Client Matching

Сопоставлять документ с профилем клиента по признакам:

- ОКВЭД;
    
- отрасль;
    
- ВЭД;
    
- онлайн-платежи;
    
- персональные данные;
    
- наличные операции;
    
- риск-профиль;
    
- размер бизнеса;
    
- тип операций.
    

Результат должен отвечать на вопрос:

```md
Почему именно этот клиент затронут?
```

Пример объяснения:

```md
Клиент релевантен, потому что у него включены признаки foreign_trade и online_payments, а документ касается валютного контроля и процедур подтверждения операций.
```

## RAG-chat

Сделать RAG, который отвечает только по базе документов.

Обязательные правила:

- retrieval по чанкам;
    
- ответ с источниками;
    
- ссылка на документ/фрагмент;
    
- режим “нет данных”;
    
- запрет на юридические советы;
    
- разделение ответа для юриста банка и для клиента;
    
- сохранение chat history.
    

Минимальные вопросы для демо:

- Какие изменения имеют высокий impact?
    
- Почему этот документ получил score 84?
    
- Каких клиентов он затрагивает?
    
- Почему ООО “ИмпортТех” релевантно?
    
- Что банк должен проверить?
    
- Сформируй уведомление клиенту простым языком.
    
- Покажи источники, на которых основан вывод.
    

## Embeddings & Retrieval

Рекомендуемая реализация:

- chunking по смысловым блокам;
    
- хранение чанков в PostgreSQL + pgvector;
    
- metadata filtering;
    
- hybrid search при наличии времени;
    
- rerank простой эвристикой или LLM rerank;
    
- top-k retrieval;
    
- source citations.
    

Важно: качество RAG важнее количества документов.

## AI Safety & Anti-Hallucination

Обязательные ограничения:

- если нет источника — писать “нет данных в базе”;
    
- не давать юридическую консультацию;
    
- не генерировать официальные документы;
    
- не придумывать штрафы;
    
- не придумывать даты;
    
- не делать категоричных выводов без evidence fragments;
    
- все критичные утверждения должны ссылаться на source fragments.
    

## Prompt Management

Промпты хранить отдельно:

```md
prompts/
- document_analysis.md
- impact_assessment.md
- client_matching.md
- notification_generation.md
- rag_answer.md
```

У каждого промпта:

- версия;
    
- назначение;
    
- входные данные;
    
- ожидаемая схема;
    
- ограничения;
    
- примеры хорошего ответа.
    

## Evaluation

Сделать минимальный eval-набор:

- 5 документов;
    
- 4 клиента;
    
- ожидаемые темы;
    
- ожидаемые impacted clients;
    
- expected impact range;
    
- вопросы для RAG;
    
- expected refusal/no-data cases.
    

Нужна не академическая оценка, а проверка, что демо не разваливается.

- автономного “юридического агента”;

- обучение собственной модели;
    
- бесконтрольные LLM-вызовы без логов;
    
- ответы без источников;

## Главный результат

AI-контур должен доказать: RegRadar не просто пересказывает документы, а превращает их в объяснимую оценку влияния, релевантность для конкретных клиентов и безопасные уведомления с опорой на первоисточники.
