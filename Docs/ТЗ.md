# Backend / Ingestion / Data Platform

## Ультра-краткое описание проекта

RegRadar — платформа, которая автоматически находит регуляторные изменения, превращает документы в структурированные карточки риска и помогает банку понять, каких бизнес-клиентов нужно предупредить.

## Краткое summary для исполнителя

Твоя зона — сделать надежный backend-контур: источники, загрузка документов, парсинг, хранение, API, статусы обработки, интеграции и основу для end-to-end demo. Продукт должен выглядеть не как мокап, а как рабочая система с реальными документами, логами и воспроизводимой обработкой.

## Важная сноска для программиста

Работать в **spec-ops подходе для вайб-кодинга**: сначала спецификации, контракты, схемы и acceptance-критерии, потом код. Обязательно подключить coding-agent модель **MiMo V2.5 / MiMo V2.5-Pro** или совместимую MiMo V2.5-модель для генерации, ревью и рефакторинга кода. Не “вайбить” архитектуру из головы: каждый крупный модуль должен иметь краткий spec, тестовый сценарий и понятный контракт.

## Роль в проекте

Ты отвечаешь за технический фундамент продукта:

- backend API;
    
- ingestion документов;
    
- парсинг источников;
    
- загрузку пользовательских файлов;
    
- хранение данных;
    
- статусы обработки;
    
- интеграцию с AI-модулем;
    
- интеграцию с Bitrix/notification adapter;
    
- Docker-запуск.
    

## Главная цель

Сделать backend, который позволяет показать полный сценарий:

1. система получает документ из источника или через upload;
    
2. извлекает текст и метаданные;
    
3. сохраняет документ и версии;
    
4. отправляет текст в AI-пайплайн;
    
5. получает structured result;
    
6. создает карточку регуляторного изменения;
    
7. показывает затронутых клиентов;
    
8. готовит уведомление;
    
9. отправляет или имитирует отправку в Bitrix.
    

## Технический стек

Рекомендуемый стек:

- Backend: **FastAPI**
    
- DB: **PostgreSQL**
    
- Vector extension: **pgvector**
    
- ORM: **SQLAlchemy / SQLModel**
    
- Migrations: **Alembic**
    
- Workers: **Celery + Redis** или FastAPI background tasks, если нужно упростить
    
- PDF: **PyMuPDF / pdfplumber**
    
- DOCX: **python-docx**
    
- OCR: внешний OCR adapter или Tesseract/PaddleOCR как fallback
    
- Containers: **Docker Compose**
    
- API docs: OpenAPI из FastAPI
    

## Основные модули

### 1. API Core

Сделать REST API для фронта и AI-модуля.

Ключевые области API:

- документы;
    
- источники;
    
- регуляторные события;
    
- клиенты;
    
- impact results;
    
- уведомления;
    
- чат-сессии;
    
- обработка файлов;
    
- health/status.
    

API должен быть понятным, стабильным и документированным через OpenAPI.

### 2. Ingestion Sources

Реализовать гибридный ingestion:

- реальный парсинг 1–2 источников;
    
- ручная загрузка файла;
    
- seed/demo data как fallback.
    

Приоритет источников:

1. Банк России;
    
2. regulation.gov.ru;
    
3. pravo.gov.ru;
    
4. пользовательский upload PDF/DOCX/TXT.
    

Для каждого документа фиксировать:

- source;
    
- original_url;
    
- title;
    
- publication_date;
    
- regulator;
    
- document_type;
    
- status;
    
- raw_html/raw_file path;
    
- text hash;
    
- processing status.
    

### 3. Document Processing

Сделать пайплайн обработки:

```md
download/upload
→ extract text
→ normalize text
→ split into chunks
→ save document
→ save version
→ send to AI pipeline
→ store structured result
```

Важно:

- дедупликация по hash;
    
- сохранение оригинального источника;
    
- обработка ошибок;
    
- повторный запуск обработки;
    
- логирование статусов.
    

### 4. Database

Минимальные сущности:

- Source
    
- Document
    
- DocumentVersion
    
- DocumentChunk
    
- RegulatoryEvent
    
- ClientProfile
    
- ClientImpact
    
- Notification
    
- ProcessingJob
    
- LLMCallLog
    
- AuditLog
    

База должна быть нормальной, но без переусложнения. Важно, чтобы можно было объяснить архитектуру жюри.

### 5. Client Profiles

Сделать CRUD/seed для профилей компаний.

Минимальные поля:

- company_name;
    
- okved;
    
- industry;
    
- size;
    
- has_foreign_trade;
    
- uses_online_payments;
    
- handles_personal_data;
    
- cash_operations_level;
    
- risk_profile;
    
- bank_segment.
    

Нужно подготовить 4–5 демо-клиентов:

- интернет-магазин;
    
- импортер;
    
- ресторанная сеть;
    
- IT/SaaS-компания;
    
- компания с большим объемом наличных операций.
    

### 6. Notification Adapter

Сделать универсальный adapter:

```md
RegRadar notification
→ channel adapter
→ Bitrix webhook / mock / log
```

В MVP достаточно:

- endpoint `send_to_bitrix`;
    
- хранение статуса отправки;
    
- лог payload;
    
- mock mode;
    
- возможность подключить реальный webhook через env.
    

Webhook URL нельзя хранить в коде или фронте.

### 7. Integration with AI Module

Backend не должен напрямую решать AI-логику. Он должен:

- передавать документ/чанки в AI-сервис;
    
- принимать structured JSON;
    
- валидировать результат;
    
- сохранять результат;
    
- отдавать фронту готовые карточки.
    

AI-вызовы должны быть логируемыми:

- provider;
    
- model;
    
- prompt version;
    
- input size;
    
- output;
    
- status;
    
- error;
    
- latency.
    

### 8. Reliability & Demo Fallback

Обязательно предусмотреть fallback:

- seed documents;
    
- заранее обработанные результаты;
    
- возможность перезапустить job;
    
- mock Bitrix;
    
- mock OCR;
    
- clear processing statuses.
    

На демо ничего не должно зависеть от одного внешнего API.

## Что должно быть готово к концу

- Docker Compose запускает backend, DB, Redis/worker при необходимости.
    
- Можно загрузить документ вручную.
    
- Можно получить документы из хотя бы одного реального источника.
    
- Документ проходит обработку и сохраняется.
    
- Фронт получает карточки изменений через API.
    
- Есть профили клиентов и связи “изменение → затронутые клиенты”.
    
- Есть endpoint для уведомления в Bitrix/mock.
    
- Есть OpenAPI-документация.
    
- Есть seed data для стабильного демо.
    

## Критерии качества

- Пайплайн воспроизводим.
    
- Ошибки не ломают всю систему.
    
- Все внешние секреты через `.env`.
    
- Есть базовые тесты на ключевые сервисы.
    
- Есть README с запуском.
    
- Есть понятные статусы обработки.
    
- Есть audit/log trail для объяснения жюри.
    

## Не делать в первую очередь

- сложную микросервисную архитектуру;
    
- идеальный парсинг всех регуляторов;
    
- полноценный legal diff;
    
- сложные роли и права;
    
- enterprise security beyond MVP;
    
- ручные костыли без сохранения в БД.
    

## Главный результат

Backend должен доказать: RegRadar — это не презентация, а рабочая система, которая принимает регуляторный документ, превращает его в структурированное событие и доводит его до клиента через понятный продуктовый контур.
